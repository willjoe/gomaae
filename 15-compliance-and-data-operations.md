# Compliance & Data Operations (GDPR/CCPA)

In a High-Integrity architecture where engineers have strictly no standing access to production databases, handling compliance requests—specifically the "Right to be Forgotten" (GDPR) or data deletion requests (CCPA)—requires a carefully orchestrated mix of self-service automation and rigorous, cryptographically verified deployment pipelines.

This document outlines how data deletion and anonymization are executed without compromising the High-Integrity data perimeter.

---

## 1. Automated User-Driven Deletion

The primary defense against compliance bottlenecks is empowering the end-user to manage their own data via automated, self-service mechanisms built directly into the application.

### Self-Service PII Deletion
*   Users must have access to a "Delete My Account" or "Manage My Data" feature within the application interface.
*   When triggered, the application backend automatically executes the deletion or anonymization of their isolated Personally Identifiable Information (PII).

### Shared Data Handling (e.g., Chat Logs)
*   Data that involves multiple users (like a direct message history or collaborative document) is treated as **Shared Data**.
*   If User A requests deletion, the system soft-deletes or cryptographically orphans User A's association with the data. 
*   **Result:** User A loses all access and their PII is scrubbed (e.g., their name becomes "Deleted User"), but the underlying shared data remains available to User B to maintain application integrity.

---

## 2. The High-Integrity Query Pipeline (Manual Data Operations)

When data deletion cannot be handled via automated self-service (e.g., scrubbing PII from an external BI tool, removing anomalous data corruption, or handling complex bespoke compliance requests), an engineer must manually intervene.

Because **no engineer** (Data Engineer, DBA, or Analyst) has raw `DELETE` access to production, manual data operations must pass through the **High-Integrity Query Pipeline**.

### Step 1: The Deletion Ticket & PR Submission
The engineer (typically a Data Engineer or Database Admin) is assigned a compliance ticket. They write the deletion script and submit a Pull Request.

The PR must contain three specific SQL scripts (or equivalent ORM logic):
1.  **The Check Query (Pre):** A `SELECT` query that identifies exactly what rows will be affected.
2.  **The Mutation Query:** The actual `DELETE`, `UPDATE` (Soft Delete), or masking execution.
3.  **The Verification Query (Post):** A follow-up query to prove the mutation was successful.

### Step 2: The Transaction-Rollback Test
Before human reviewers even look at the PR, the CI/CD pipeline intercepts the code and runs a **Transaction-Rollback Test** against the live production database.

1.  The pipeline opens a database transaction.
2.  It executes the **Check Query** and logs the output (e.g., "5 rows found matching User ID X").
3.  It executes the **Mutation Query**.
4.  It executes the **Verification Query** and compares the difference (e.g., "0 rows found matching User ID X").
5.  **Crucial Step:** The pipeline immediately executes a `ROLLBACK`. The production data is untouched.
6.  The pipeline appends the result of the simulation (the diff) directly to the Pull Request for reviewers to inspect.

### Step 3: Review and Execution
*   Reviewers (including Security Engineers and Data Architects) inspect the automated diff generated by the rollback test. They can mathematically verify that the query behaves exactly as intended without seeing the raw data payload.
*   Once all required reviewers approve the PR, the code is merged.
*   **Execution:** The CI/CD pipeline (which holds the highly guarded `db_admin` JIT credentials) automatically connects to the production database and executes the approved query for real, permanently deleting or masking the requested data.
